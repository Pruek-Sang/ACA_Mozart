## 🔄 2. Workflow Integration

### การทำงานของ Session & State
1. **Start Session**: User เข้าเว็บ → เรียก `/api/v1/session/start` → สร้าง record ใน `mozart.sessions`
2. **Conversation**: User คุย/ตอบคำถาม → Update `messages` / `site_context` ใน DB
3. **Keepalive**:
   - Supabase Free Tier จะ pause ถ้าไม่มี traffic 7 วัน
   - เราแก้โดยใส่ **Ping Query** ไว้ใน `/` (Health Check)
   - ทุกครั้งที่ GitHub Actions deploy หรือ Health check ทำงาน → Supabase ตื่น

### การทำงานของ Rate Limiter
- **Limit:** 10 requests/นาที สำหรับ `/design` (Expensive Endpoint)
- **Limit:** 20 requests/นาที สำหรับ `/ask`
- **Logic:** ใช้ In-memory counter (Simple & Fast)
- **Fallback:** ถ้า Module โหลดไม่ได้ → ข้ามการเช็ค (Allow all)

## ⚠️ 4. Risk Assessment & Code Review Findings

จากการตรวจสอบ Code ล่าสุด พบจุดที่ต้องระวัง (Work in Progress):

### 🔴 Risk 1: IP Address behind Load Balancer
- **Code:** `request.client.host`
- **Issue:** บน Cloud Run, IP ที่เห็นอาจเป็นของ Load Balancer ไม่ใช่ User จริง
- **Result:** ถ้า Rate Limit ทำงาน อาจจะ Block ผู้ใช้ทุกคนพร้อมกัน
- **Fix:** ต้องเปลี่ยนไปอ่าน Header `X-Forwarded-For` หรือ `X-Real-IP` ในอนาคต
- **Mitigation:** ตอนนี้ Limit ไว้ที่ 10/นาที ซึ่งสูงพอสมควรสำหรับ MVP แต่ต้องแก้ก่อน Scaling

### 🟡 Risk 2: Supabase Health Check Latency
- **Code:** `root()` ทำ query ไป DB
- **Issue:** ถ้า Supabase ช้ามากๆ Health check อาจ timeout และ Cloud Run อาจ restart container
- **Mitigation:** ใส่ `try-except` ครอบไว้แล้ว ถ้า DB พัง Service ยังตอบ 200 OK (Alive) แต่แจ้ง status error แทน (ไม่ Crash)

### 🟢 Risk 3: Dependency Conflict
- **Issue:** กลัว `supabase-py` ตีกับ Websocket หรือ libs อื่น
- **Check:** ใช้ v2.0.0+ ซึ่งทำงานแบบ HTTP REST (Stateless) ไม่เปิด Websocket connection ค้างไว้ → **Safe**
